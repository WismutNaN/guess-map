## 1. Суть продукта

Нужно desktop-приложение для тренировки GeoGuessr в формате **интерактивного гео-атласа подсказок**.

Приложение должно позволять:

* смотреть мир и отдельные страны/регионы на быстрой интерактивной карте;
* включать и комбинировать разные типы подсказок;
* видеть различия **сразу на карте**, а не только в боковой карточке;
* пополнять базу знаний **через интерфейс**, без ручного поиска ID регионов;
* привязывать знания как к странам, так и к внутренним регионам;
* при необходимости смещать визуальные подсказки относительно центра региона через координатный anchor / offset.

## 2. Базовая продуктовая идея

Это не GIS-редактор границ.
Границы пользователь **не редактирует**.

Нужна система, где:

* есть готовые геометрии стран и внутренних регионов;
* пользователь выбирает регион прямо на карте или через поиск/список;
* затем назначает ему свойства, подсказки, изображения, пометки и приоритеты отображения;
* результат сразу виден на карте в тех слоях, которые включены.

То есть главный редактор — это **редактор атрибутов и привязок**, а не редактор геометрии.

## 3. Основные пользовательские сценарии

### 3.1. Режим изучения

Пользователь включает несколько слоёв одновременно и видит различия:

* флаги;
* сторона движения;
* образцы письменности;
* телефонные региональные особенности;
* road markings;
* знаки;
* столбы / bollards;
* coverage / camera meta.

### 3.2. Режим сравнения

Пользователь приближает страну или регион и сразу видит, чем соседние области отличаются друг от друга.

Примеры:

* Индия: различия по регионам через примеры письменности;
* США: телефонные региональные подсказки;
* Бразилия: региональные телефонные или иные локальные мета-признаки;
* страны с левосторонним/правосторонним движением — цветом по полигонам.

### 3.3. Режим наполнения базы знаний

Пользователь:

* ищет регион по названию;
* кликает его на карте;
* назначает свойства через форму;
* может выделить несколько регионов и массово применить одну подсказку;
* прикрепляет изображение-пример;
* сохраняет изменения;
* сразу проверяет визуализацию на карте.

## 4. Обязательные функциональные требования

### 4.1. Карта

* очень быстрая интерактивная карта;
* плавный pan/zoom;
* работа с несколькими слоями подсказок одновременно;
* отображение стран и внутренних регионов;
* отображение крупных городов;
* поддержка нескольких провайдеров подложки;
* перспективное подключение Google basemap как опционального провайдера.

Технический комментарий: для такого сценария разумная база — **Tauri 2 + TypeScript UI + MapLibre GL JS**. MapLibre рендерит интерактивные карты через WebGL, работает со `sources` и `layers`, поддерживает data-driven styling через expressions, а Tauri даёт типобезопасный вызов Rust-команд из фронтенда. Для локальных ресурсов и базы пригодны официальные Tauri plugins для файловой системы и SQL. ([Tauri][1])

### 4.2. Слои подсказок

Нужны переключаемые слои минимум следующих типов:

* flags;
* driving side;
* scripts / writing clues;
* phone clues;
* coverage;
* camera generation / meta;
* road markings;
* signs;
* poles / bollards;
* произвольные текстовые заметки.

### 4.3. Комбинирование слоёв

Пользователь должен иметь возможность включать несколько типов подсказок одновременно.
Например:

* флаги + образцы письменности;
* флаги + письменность + phone clue;
* только сторона движения;
* только знаки и разметка.

### 4.4. Региональная детализация

Подсказки должны поддерживать разные уровни привязки:

* страна;
* внутренний регион;
* тематический регион внутри страны;
* точка / anchor по координатам.

Это нужно, чтобы не ограничиваться только admin-границами. Например, если геометрия региона подходит, данные вешаются на регион. Если геометрия слишком грубая, объект отображения можно сместить через отдельный anchor point.

### 4.5. Интерактивное наполнение знаний

Через UI пользователь должен мочь:

* выбрать регион на карте;
* увидеть, какие свойства уже назначены;
* добавить новое свойство;
* прикрепить изображение;
* указать краткое и полное описание;
* массово применить свойство к группе выбранных регионов;
* редактировать/удалять привязки;
* сохранять изменения без ручной работы с ID регионов.

## 5. Что именно НЕ требуется

* не нужен редактор границ;
* не нужен сложный GIS-workflow по вершинам, snapping, topology;
* не требуется рисование новых полигонов в первой версии;
* письменность можно показывать **изображениями и скриншотами**, а не обязать движок красиво рендерить все письменности как текст.

Это сильно упрощает архитектуру. Значит, основной упор надо делать на:

* хорошую модель данных;
* интерфейс выбора регионов;
* пакетное назначение атрибутов;
* быстрый рендер.

## 6. Как должна выглядеть модель данных

Рекомендую разделить данные на 5 сущностей.

### 6.1. Region

Описывает геометрию и идентичность региона.

Поля:

* `id`
* `name`
* `country_code`
* `region_level` (`country`, `admin1`, `admin2`, `theme_region`)
* `geometry_ref`
* `anchor_lng`
* `anchor_lat`
* `anchor_offset_x`
* `anchor_offset_y`
* `priority`
* `is_active`

### 6.2. HintType

Справочник типов подсказок.

Поля:

* `id`
* `code` (`flag`, `driving_side`, `script_sample`, `phone_hint`, `road_marking`, `sign`, `pole`, `bollard`, `coverage`, `camera_meta`, `note`)
* `title`
* `display_family`
* `default_icon`

### 6.3. RegionHint

Конкретная подсказка, привязанная к региону.

Поля:

* `id`
* `region_id`
* `hint_type_id`
* `short_value`
* `full_value`
* `image_asset_id`
* `icon_asset_id`
* `color`
* `sort_order`
* `min_zoom`
* `max_zoom`
* `is_visible`
* `confidence`
* `source_note`

### 6.4. Asset

Изображения и иконки.

Поля:

* `id`
* `file_path`
* `kind` (`flag`, `sample`, `icon`, `thumbnail`)
* `width`
* `height`
* `caption`

### 6.5. RevisionLog

История правок.

Поля:

* `id`
* `entity_type`
* `entity_id`
* `action`
* `created_at`
* `comment`

## 7. Как должна работать визуальная привязка данных

Главная мысль: пользователь не должен искать region ID.

Нужен такой workflow:

1. пользователь выбирает активный слой знаний;
2. ищет страну/регион через поиск или кликом на карте;
3. выбранные регионы подсвечиваются;
4. справа открывается панель свойств;
5. пользователь назначает подсказку;
6. карта тут же обновляется.

Дополнительно нужно:

* multi-select нескольких регионов;
* массовое назначение одной подсказки;
* фильтр “показывать только регионы текущей страны”;
* список уже выбранных регионов;
* кнопка “применить ко всем выбранным”.

## 8. Требования к отображению на карте

### 8.1. Общий принцип

Карта должна показывать **подсказки прямо на карте**, но без визуального шума.

Поэтому надо разделить визуализацию по масштабам:

**Мелкий масштаб**

* цветовая заливка полигонов;
* минимальные иконки;
* короткие подписи.

**Средний масштаб**

* больше региональных различий;
* несколько подсказок рядом;
* более детальные подписи.

**Крупный масштаб**

* мини-примеры;
* всплывающие карточки;
* pinned tooltip / popup;
* несколько изображений и заметок.

### 8.2. Driving side

Нужна окраска полигонов по стороне движения:

* left;
* right;
* unknown / mixed.

Технический комментарий: MapLibre style spec позволяет задавать визуальные свойства слоя на основе атрибутов объектов через data expressions и управлять плотностью/видимостью через zoom-expressions. Для заливки по `driving_side` это прямой и штатный сценарий. ([MapLibre][2])

### 8.3. Одновременное отображение нескольких подсказок

Для каждого региона желательно иметь предсказуемые “слоты” вокруг anchor:

* сверху — флаг;
* справа — письменность;
* снизу — phone clue;
* слева — специальная мета-подсказка;
* центр — подпись региона или главный символ.

Технический комментарий: в MapLibre для этого есть нужные механизмы symbol-layers: `symbol-sort-key`, `text-variable-anchor`, `icon-offset`, `text-offset`, `icon-allow-overlap`, `text-allow-overlap`. Это позволяет управлять приоритетом, коллизиями и альтернативным размещением подписи. ([MapLibre][3])

### 8.4. Важное ограничение по UX

На карте не надо пытаться всегда показывать большие карточки или много HTML-элементов.
Основной рендер должен быть через map-native слои и символы, а popup / HTML-карточки — только для:

* hover;
* pin;
* активного выбранного региона.

Это уменьшит визуальный шум и упростит производительность.

## 9. Города

Нужно отображать хотя бы крупные города.

Рекомендуемая логика:

* на мировом масштабе — столицы и крупнейшие города;
* при приближении — больше городов;
* опция отключения city labels, если они мешают обучающим слоям.

Технический комментарий: для первой версии подойдут готовые наборы населённых пунктов. Natural Earth даёт populated places с именами, столицами, major cities и scale rank для фильтрации подписей; если позже понадобится более плотная база, GeoNames покрывает все страны и содержит более 11 млн географических названий. ([Natural Earth][4])

## 10. Провайдеры карт и подложка

Нужно сразу заложить архитектуру с несколькими провайдерами подложки.

### 10.1. Что требуется

* абстракция `BasemapProvider`;
* возможность переключения провайдера без переделки логики слоёв знаний;
* отдельные настройки и атрибуция для каждого провайдера.

### 10.2. Рекомендуемая стратегия

**По умолчанию** использовать открытый векторный basemap.
**Google basemap** — как опциональный провайдер второй очереди.

Технический комментарий: OpenMapTiles позволяет self-host world map и генерировать vector tiles из OSM-тегов и собственных geodata. Google Map Tiles API действительно рассчитан и на кастомные desktop/non-web приложения и даёт 2D tiles для Roadmap / Satellite / Terrain, но требует API key/session token и накладывает жёсткие условия по атрибуции, отсутствию скрытия логотипа и ограничениям на prefetch/cache/storage/offline use. Поэтому Google лучше закладывать как расширение, а не как первую обязательную зависимость. ([OpenMapTiles][5])

## 11. Источники геометрии регионов

Поскольку границы редактировать не нужно, надо брать готовые геометрии и строить работу вокруг них.

Для первой версии достаточно:

* страны;
* admin-1 регионы;
* при необходимости тематические наборы регионов как отдельные слои.

Технический комментарий: Natural Earth даёт admin-1 states/provinces и populated places; при этом у admin-1 есть ограничения для части мелких стран, а сами границы в Natural Earth по умолчанию даны как de facto. Это нормально для учебного атласа, но важно зафиксировать как осознанное решение. ([Natural Earth][6])

## 12. Хранение данных

Рекомендация:

* **SQLite** — источник истины для regions, hints, assets, revision log;
* изображения — файлово;
* compiled map layers — отдельно.

Технический комментарий: официальный Tauri SQL plugin поддерживает SQLite, а FS plugin — управляемую работу с файлами и base directories. Это хороший вариант для локального knowledge base приложения. ([Tauri][7])

## 13. Формат быстрой картографической доставки

Для быстрого рендера поверх карты стоит разделить:

* редактируемые исходные данные;
* compiled слои для отображения.

Рекомендуемая схема:

* source of truth: SQLite + GeoJSON/табличные справочники;
* runtime representation: vector tiles / PMTiles.

Технический комментарий: PMTiles рассчитан на прямое чтение в MapLibre, в том числе для thematic overlay tilesets и basemap tilesets; интеграция делается через `pmtiles` JS-библиотеку и `addProtocol`. В Tauri можно отдельно бандлить дополнительные файлы как resources, если нужно поставлять локальные архивы тайлов или ассеты вместе с приложением. ([Protomaps Docs][8])

## 14. Требования к редактору знаний

Нужен отдельный экран или режим **Knowledge Editor**.

### 14.1. Обязательные элементы

* дерево/список слоёв подсказок;
* поиск по регионам;
* выбор региона на карте;
* мультивыбор;
* панель свойств;
* загрузка изображения;
* предварительный просмотр отображения;
* сохранение;
* журнал изменений.

### 14.2. Массовые операции

* назначить один hint всем выбранным регионам;
* назначить общую иконку;
* выставить общие zoom-правила;
* применить общий приоритет отображения.

### 14.3. Привязка через точку

Если центр региона неудобен, нужно иметь возможность задать:

* anchor point;
* визуальный offset.

Это особенно полезно для маленьких, вытянутых или визуально перегруженных регионов.

## 15. Режимы отображения для пользователя

Нужно минимум 4 пресета плотности:

* `Minimal`
* `Balanced`
* `Dense`
* `Study`

И 3 режима представления подсказок:

* `Icons only`
* `Icons + short text`
* `Icons + thumbnails`

Это позволит избежать перегрузки карты.

## 16. Производительность и отладка

Нефункциональные требования:

* карта должна оставаться отзывчивой при большом количестве регионов и подсказок;
* нельзя бесконтрольно увеличивать количество тяжёлых слоёв и DOM-оверлеев;
* надо ограничивать видимость слоёв по min/max zoom;
* нужно упростить стили для массовых слоёв;
* нужно иметь внутренние debug-инструменты для анализа коллизий и тайлов.

Технический комментарий: в документации MapLibre есть отдельные рекомендации для больших GeoJSON-наборов — упрощать стиль, ограничивать min/max zoom и не пытаться рендерить лишнюю детализацию на малых масштабах. Для диагностики полезны встроенные debug-переключатели вроде `showCollisionBoxes` и `showTileBoundaries`. ([MapLibre][9])

## 17. Рекомендуемый стек

Рекомендуемый стек для первой версии:

* **Tauri 2**
* **Rust**
* **TypeScript**
* **React** или **Vue**
* **MapLibre GL JS**
* **SQLite**
* **PMTiles** для compiled overlay layers
* открытый basemap по умолчанию
* Google basemap как будущий optional provider

Технический комментарий: такой стек хорошо подходит именно под desktop-atlas с локальной базой знаний, быстрым рендером и возможностью расширять картографические провайдеры без смены всей архитектуры. ([Tauri][1])

## 18. Что стоит явно зафиксировать в ТЗ как архитектурные решения

1. **Приложение viewer-first, а не GIS-editor.**
   Границы не редактируются.

2. **Назначение знаний должно идти визуально через UI.**
   Никакого ручного поиска ID.

3. **Основа модели — Region + Hint + Asset + Anchor.**

4. **Подсказки должны работать на нескольких уровнях детализации.**

5. **Карта должна поддерживать несколько basemap providers.**

6. **Google basemap не делать обязательным в первой версии.**

7. **Визуализация должна быть zoom-dependent.**

8. **Основной рендер — через map layers, а не через множество HTML-маркеров.**

9. **Хранилище знаний — локальное и редактируемое пользователем.**

10. **Нужна история изменений.**

## 19. Что стоит исключить из первой версии

Чтобы не расползся объём:

* не делать редактор границ;
* не делать сложную топологию;
* не делать полноценный конструктор новых тематических полигонов;
* не делать сразу полный мировой контент по всем clue-типам;
* не делать сразу Google-зависимую архитектуру.

## 20. Формулировка задачи для архитектора/разработчика

Можно передать ему так:

> Нужно спроектировать desktop-приложение для обучения GeoGuessr в виде интерактивного атласа подсказок. Основа — быстрая карта мира и регионов, переключаемые и комбинируемые слои знаний, привязка свойств к готовым регионам без редактирования границ, а также отдельный интерфейс для наполнения базы знаний через визуальный выбор регионов. Приложение должно поддерживать цветовую заливку регионов, иконки, короткие подписи, изображения-примеры, anchor-точки, историю изменений, крупные города и несколько провайдеров подложки. Первая версия должна быть ориентирована на производительность и удобство авторинга знаний, а не на GIS-редактирование геометрии.
